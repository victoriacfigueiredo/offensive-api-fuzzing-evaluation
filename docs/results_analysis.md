# Comparative Analysis of Fuzzing Results

## Overview

Three API fuzzing tools were evaluated against the OWASP crAPI application:

* Schemathesis
* RESTler
* EvoMaster

Although all three tools use the OpenAPI specification as an input, they follow different testing strategies and therefore produced different results.

The objective of this analysis is not only to compare the number of vulnerabilities discovered, but also to understand the strengths and limitations of each approach.

---

## Quantitative Results

| Tool         | Average Coverage (%) | Average Requests / Tests | Average Findings | Average Server Errors |
| ------------ | -------------------: | -----------------------: | ---------------: | --------------------: |
| Schemathesis |                100.0 |                   4622.0 |            120.0 |                  15.7 |
| RESTler      |                 49.6 |                    273.7 |             22.3 |                  22.3 |
| EvoMaster    |                 50.0 |                   7611.7 |             77.3 |                  77.3 |

### Coverage

Schemathesis achieved complete OpenAPI coverage, successfully exercising every operation described in the specification.

RESTler and EvoMaster covered approximately half of the specification. This lower coverage is partially explained by their reliance on valid resource creation sequences and application state transitions.

### Request Volume

EvoMaster generated the highest number of test cases, exceeding 7,600 tests on average.

Schemathesis generated approximately 4,600 test cases.

RESTler generated significantly fewer requests, focusing on dependency-aware request sequences rather than brute-force exploration.

### Findings

Schemathesis reported the largest number of findings overall.

However, many of these findings correspond to specification violations, undocumented responses, authentication inconsistencies, and state-related anomalies.

EvoMaster generated fewer findings than Schemathesis but considerably more than RESTler, mainly due to its aggressive search-based exploration strategy.

RESTler produced the smallest number of findings but identified several issues involving resource dependencies and state transitions.

# Confirmed Vulnerabilities

## Findings Identified by Schemathesis

### Authentication Bypass

**Endpoint**

```
/workshop/api/shop/orders/{order_id}
```

**Description**

Order information could be accessed without authentication.

**Impact**

Unauthenticated users could retrieve order details and associated information.

---

### IDOR / Broken Access Control

**Endpoint**

```
/workshop/api/shop/orders/{order_id}
```

**Description**

Changing the order identifier allowed access to other users' orders.

**Impact**

Unauthorized access to sensitive order information.

---

### Improper Authentication Handling

**Endpoint**

```
/identity/api/v2/user/dashboard
```

**Description**

Instead of returning HTTP 401 Unauthorized, the endpoint returned HTTP 404 and a user-not-found message.

**Impact**

Authentication failures were handled incorrectly and leaked implementation details.

---

### Improper Error Handling

**Endpoints**

```
/identity/api/auth/v2/check-otp
/identity/api/auth/v3/check-otp
/identity/api/auth/v2.7/user/login-with-token
/workshop/api/mechanic/mechanic_report
```

**Description**

Malformed requests triggered HTTP 500 Internal Server Error responses.

**Impact**

Server-side exceptions were exposed instead of controlled client-side validation errors.

---

# Findings Identified by EvoMaster

EvoMaster reported the highest number of potential faults.

Most findings were generated by:

* InvalidValueChecker
* InvalidDynamicObjectChecker
* Main Driver exploration

Common issues included:

* HTTP 500 responses
* HTTP 503 responses
* Failures caused by malformed values
* Failures caused by invalid resource identifiers
* Failures triggered during state transitions

Many of these findings overlap conceptually with RESTler's results, although EvoMaster produced them in significantly larger quantities.

---

# Common Findings

The following issues were independently identified by multiple tools.

| Finding                                            | Schemathesis | RESTler | EvoMaster |
| -------------------------------------------------- | ------------ | ------- | --------- |
| OTP validation returns HTTP 500                    | ✓            | ✓       | ✓         |
| Token-based login returns HTTP 500                 | ✓            | ✓       | ✓         |
| Invalid coupon processing triggers server errors   | ✗            | ✓       | ✓         |
| Email verification errors trigger HTTP 500         | ✗            | ✓       | ✓         |
| Invalid mechanic-related requests trigger HTTP 500 | ✓            | ✓       | ✓         |

These overlaps increase confidence that the issues correspond to genuine application weaknesses rather than tool-specific false positives.

# Unique Findings

## Schemathesis Only

* Authentication Bypass
* IDOR / Broken Access Control
* Improper Authentication Handling

These findings were not identified by RESTler or EvoMaster.

This suggests that Schemathesis is particularly effective at discovering authorization and specification-related issues.

---

## RESTler Only

* Resource dependency failures
* Invalid dynamic object handling
* State-transition-related failures

These findings leverage RESTler's dependency-aware request sequencing capabilities.

---

## EvoMaster Only

* Large numbers of automatically generated fault-inducing inputs
* Extensive invalid-value exploration
* Search-based discovery of edge cases

This behavior reflects EvoMaster's evolutionary testing strategy.

---

# Interpretation

The results suggest that the three tools complement one another rather than compete directly.

Schemathesis was the most effective at identifying authorization issues and specification inconsistencies.

RESTler excelled at exercising endpoint dependencies and application workflows.

EvoMaster generated the largest number of fault-inducing inputs and exposed the greatest volume of server-side failures.

A combined testing strategy would therefore provide substantially better coverage than relying on any single tool.
