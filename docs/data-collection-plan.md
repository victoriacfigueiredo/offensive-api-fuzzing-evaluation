# Data Collection Plan

## 1. Collection Objective

The objective of the data collection plan is to register and organize the data obtained during the execution of the fuzzing tools, enabling the comparison of their effectiveness in detecting vulnerabilities in REST APIs.

---

## 2. Collected Data

During the experiments, the following data will be collected:

- Number of detected vulnerabilities;
- Types of identified vulnerabilities;
- Number of unexpected HTTP responses;
- Number of HTTP 5xx responses;
- Endpoint coverage achieved;
- Total execution time;
- Logs generated by the tools;
- Automatically generated reports;
- Number of false positives;
- Evidence of authentication and authorization flaws;
- Evidence of input validation flaws;
- Payloads that caused insecure behavior.

---

## 3. Collection Method

The data will be collected through:

- Reports automatically generated by the fuzzing tools;
- HTTP logs recorded during testing;
- Analysis of API responses;
- Manual observation of relevant failures;
- Monitoring Docker containers during the experiments.

---

## 4. Data Storage

The collected data will be stored in:

- CSV files containing quantitative metrics;
- Tool-generated log files;
- Reports generated by the fuzzing tools;
- Records organized by evaluated tool and API.

The data will be organized into separate directories to facilitate comparative analysis.

---

## 5. Collection Frequency

Data collection will occur after each execution of a fuzzing tool against each evaluated REST API.

Each tool/API combination may be executed multiple times to improve result consistency.

---

## 6. Data Analysis

The collected data will be analyzed in both quantitative and qualitative ways.

### Quantitative Analysis

The following metrics will be compared:

- Number of detected vulnerabilities;
- Endpoint coverage;
- Number of detected errors;
- Execution time;
- Number of false positives.

### Qualitative Analysis

The following aspects will be analyzed:

- Relevance of detected vulnerabilities;
- Ability to detect critical flaws;
- Ease of configuration and usage;
- Effectiveness in offensive security scenarios.

---

## 7. Expected Results

The research is expected to identify relevant differences between the evaluated tools, highlighting:

- Tools with higher vulnerability detection capability;
- Tools with better endpoint coverage;
- Tools more effective in detecting vulnerabilities related to the OWASP API Security Top 10;
- Main advantages and limitations of each fuzzing approach applied to REST API security testing.
